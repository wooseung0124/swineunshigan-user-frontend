# 기여 가이드

## 브랜치 규칙
- `main` 직접 push 금지 ❌
- `develop`에서 분기 후 작업
- 브랜치명: `feature/기능명`, `fix/버그명`, `hotfix/긴급수정명`

## 커밋 메시지 규칙
feat: 새로운 기능 추가  
fix: 버그 수정  
docs: 문서 수정  
style: 코드 포맷팅 (기능 변경 없음)  
refactor: 코드 리팩토링  
test: 테스트 코드  
chore: 패키지 업데이트 등 기타  

## PR 규칙
- PR 제목은 커밋 규칙과 동일하게 작성
- PR 본문에 작업 내용 간단히 설명
- 최소 1명 이상 리뷰 승인 후 머지
- 머지는 Owner가 최종 승인

## 작업 흐름
1. develop 최신화 후 브랜치 분기
2. 작업 완료 후 PR 생성 (develop 방향)
3. 리뷰 요청 → 승인 → 머지
